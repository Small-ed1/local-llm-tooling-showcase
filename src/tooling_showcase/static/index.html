Help

Setup, troubleshooting, and page guide

Use this when Ollama, the interface, tools, model routing, sessions, or local state behave unexpectedly.

Ollama and models

Use `ollama list` to confirm models are installed and `ollama serve` or your system service to expose `127.0.0.1:11434`.

If the model picker is empty, verify `/api/tags` is reachable and check the runtime strip. Benchmarked profile suggestions appear only after `tooling-showcase benchmark` has produced local results.

If responses are slow, benchmark models, lower context tokens in Settings, or select a smaller installed model.

Chat page

Use natural language first. The planner can call safe tools when a request needs files, search, retrieval, memories, or current information.

Use retry variants when an answer needs more detail, a shorter version, debugging, or a different model.

On mobile, open the sidebar with the floating menu button and keep long code blocks horizontally scrollable.

Sessions and memories

Sessions and UI memories live in browser storage. Backend memories created by model tools live under the local `state/` directory.

Use memories only for stable preferences and facts. Do not store secrets, credentials, or one-off prompt content.

If history looks wrong, use the Sessions page search, export all sessions, or clear browser UI state from Settings.

Tools and confirmations

Read/search/index tools are safe for automatic planner use. Shell execution stays confirmation-gated and destructive patterns are blocked.

If a tool is rejected, it is either hidden from the planner, unavailable in the runtime, or requires confirmation.

Use the Manual Tool Console to test exact tool arguments before expecting the model planner to call them.

Adapters and workspace

Adapters inspect nearby projects configured by `TOOLING_SHOWCASE_PORTFOLIO`. If none appear, check the environment variable and project locations.

Use tree view and file search before reading files when paths are uncertain.

Journal and debugging

The journal records routes, tool calls, model fallback behavior, and failures. Use it to verify whether a tool actually ran.

If UI state and backend state disagree, refresh runtime data, reload the page, and inspect `state/events.jsonl`.

Settings and profiles

System prompts, profile fields, theme settings, and model options are browser-local. Export important sessions before clearing browser data.

Model options are sent to Ollama per request. Unsupported options may be ignored by a model or endpoint.

Install and benchmark flow

Run `./install.sh` for guided setup. It can create a virtualenv, install the package, run tests, and offer benchmarking when several local models exist.

After installing new Ollama models, run `./install.sh` again or `tooling-showcase benchmark`; only unbenchmarked models are tested unless `--all` is used.
